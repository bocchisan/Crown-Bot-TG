# Crown Bot TG — Спецификация

**v1.0 · Недоверенный клиент · Приватный канал по репутации и подписке**

Бот-модератор приватного Telegram-канала. Два права входа, объединяемые по «или»:

- **репутация** у владельца канала не ниже порога — пропуск навсегда (книга монотонна);
- **живая подписка** — эскроу формы `stream` с резолвером канала, оплаченное окно которого покрывает «сейчас».

Читается вместе с `crown-core/docs/core-spec.md` (книга), `crown-factory/docs/factory-spec.md` §2.4 (форма `stream`) и `crown-games/subscription/docs/game-spec.md` (канистра-резолвер). Термины оттуда не переопределяются. Бот — недоверенная зона: всё в этом документе описывает **политику одного клиента**, а не гарантии платформы; гарантии остаются у эскроу, сплиттера, канистр.

Хостинг: сервис бота — VPS (long polling, публичного HTTP нет); мини-апп — asset-канистра ICP (HTTPS, сертифицированная статика). Смерть VPS останавливает только членский менеджмент: все денежные флоу идут через мини-апп напрямую в канистры и цепь.

---

## 1. Компоненты и роли

| Компонент | Где живёт | Секреты | Что делает |
|---|---|---|---|
| `core/` | библиотека в `tg/` и `miniapp/` | нет | соль/адрес эскроу, раскладки сообщений, чтение цепи/канистр, сборка транзакций |
| `tg/` | VPS | токен бота (env) | Bot API: заявки, кики, диалоги; SQLite; ревизия; напоминания |
| `miniapp/` | asset-канистра ICP | нет | кошелёк: SIWS-подписи, `create_escrow`, сбор кусков, отмена |

| Роль | Как действует |
|---|---|
| Владелец канала | добавляет бота админом, задаёт политику, собирает куски своим кошельком |
| Зритель | привязывает кошелёк SIWS-подписью, входит по репутации или подписке, отменяет своим кошельком |
| Бот | одобряет заявки и кикает — только по своим записям; транзакций не шлёт |

Один экземпляр бота обслуживает много каналов. Права в Telegram: админ канала с «одобрять заявки» и «удалять участников»; канал — приватный, вход по заявкам (join request), инвайт-ссылки ботом не раздаются.

## 2. Идентификаторы

**`channel_id`** — 32 байта, он же `subscription_id` канистры (у неё id непрозрачен; доменный префикс — наша обязанность клиента, game-spec «Подписки» §2):

```
channel_id = sha256("crown:bot-tg:v1" ‖ i64le(tg_chat_id) ‖ owner_wallet(32) ‖ u64le(setup_nonce))
```

`tg_chat_id` — числовой id канала в Telegram; `owner_wallet` — кошелёк владельца, привязанный при сетапе; `setup_nonce` — u64, выбирается ботом при сетапе. Детерминирован и восстановим из записи канала; смена владельца или пере-сетап с новым nonce = новый `channel_id` = новый резолвер = старые эскроу доживают у старого ключа. Раскладка пиннуется кросс-инструментальным вектором (python3 hashlib).

**Резолвер канала** — `subscription.get_resolver(chain, channel_id)`; берётся при сетапе, кэшируется в записи канала (детерминирован — кэш безопасен).

## 3. Привязка кошелька к Telegram-аккаунту

**Транспорт.** Кнопка мини-аппа (ReplyKeyboard `web_app`) в личке бота. Мини-апп получает челлендж в start-параметре, возвращает результат через `Telegram.WebApp.sendData` (≤4096 байт — подпись и ключ помещаются); публичный HTTP сервису не нужен. `initData` каждого веб-запуска верифицируется HMAC токена бота — Telegram сам аутентифицирует `telegram_id`.

**Челлендж.** Человекочитаемый текст фиксированной раскладки (кошельки показывают его при подписи), генерируется ботом, живёт `CHALLENGE_TTL = 300` секунд, nonce одноразов (память процесса; рестарт инвалидирует незавершённые — человек жмёт ещё раз):

```
crown-bot-tg v1
bot: <bot_username>
channel: <hex(channel_id)>
telegram: <telegram_id>
action: link
nonce: <hex(16 байт)>
expires: <unix>
```

Подпись — Ed25519 Solana-кошельком над байтами UTF-8. Бот проверяет: подпись валидна против заявленного кошелька; nonce выдан этим ботом этому `telegram_id` для этого канала и не использован; срок не истёк. Раскладка пиннуется тестом; смена раскладки = смена `v1`.

**Инварианты привязок (по каналу).**

- кошелёк → **не более одного** активного `telegram_id`. Повторная привязка занятого кошелька = перепривязка: старый аккаунт кикается, новый занимает место, кулдаун `REBIND_COOLDOWN = 72h` на следующую перепривязку этого кошелька.
- `telegram_id` → до `MAX_WALLETS = 10` кошельков.
- Привязка не удаляется при выходе из канала — только перепривязкой или отвязкой самим пользователем (отвязка тоже начинает кулдаун).

Права аккаунта = объединение прав его кошельков; складчина возможна и принята: N кошельков на одном аккаунте — одно место, не N (философия самосибила ядра, core-spec §10.1).

## 4. Закон допуска

Чистая функция от записей канала, привязок и первоисточников; в коде — один модуль, покрытый тестами на фикстурах.

```
entitled(account, channel) =
      Σ_{w ∈ wallets(account)} rep(chain, w, owner) ≥ threshold      // навсегда
   ∨  ∃ w ∈ wallets(account), ∃ escrow: subscription_alive(escrow, w, channel)
```

`rep` — `crown-index.get_reputation(chain, w, owner_wallet)`. Порог достигнут — достигнут навсегда (книга монотонна); поднятие порога владельцем не выкидывает уже впущенных: при ревизии для действующих участников применяется порог на момент их входа (хранится в записи членства).

`subscription_alive(escrow, w, channel)` — все условия сразу, по полям аккаунта эскроу из цепи:

```
escrow.resolver   == resolver(channel)            // членство
escrow.donor      == w                            // платил привязанный кошелёк
escrow.recipients == [owner_wallet]               // ровно один получатель — владелец
escrow.shares     == [10000]                      // весь кусок — владельцу
escrow.chunk      >= price(channel)               // не дешевле политики
escrow.period     == period(channel)              // ровно период политики
!escrow.settled                                   // не отменён / не исчерпан / не refund
now < t0 + n_chunks·period                        // оплаченное окно покрывает «сейчас»
```

Доступ **не зависит от `released`**: собрал владелец куски или нет — его инкассация, не членство зрителя. Схитрить нельзя: оба выхода донора (`cancel`, `refund()`) ставят `settled` — окно гаснет вместе с эскроу.

Смена цены/периода владельцем действует на новые эскроу; действующие доживают своё оплаченное окно по правилам рождения.

## 5. Жизненный цикл членства

**Сетап владельца.** Бот в канале админом → `/setup` в личке → выбор канала (бот сверяет свои права и права владельца через Bot API) → мини-апп: привязка кошелька владельца (SIWS, `action: link`) → политика: `price` (minor units USDC за период), `period` (пресеты: 7 суток, 30 суток; кастом ≥ 1 суток), `threshold` (minor units репутации; 0 = гейта нет; `price = 0` = подписки нет; оба нуля — отказ) → бот вычисляет `channel_id`, берёт резолвер у канистры, пишет запись, отдаёт диплинк `t.me/<bot>?start=<hex(channel_id)>`.

**Вход зрителя.** Диплинк → карточка канала → привязка кошелька (§3) → проверка `entitled` → бот одобряет заявку на вступление (или ведёт в мини-апп подписаться). Заявки от людей без привязки не отклоняются — висят владельцу на ручное решение, бот помечает их «не с платформы».

**Ревизия.** Периодический проход (`REVISION_PERIOD = 6h`) по записям членства: `entitled` пересчитывается из первоисточников. Потерявшему право — предупреждение в личку и grace-период (`GRACE = 72h`), после — кик и пометка записи. Участники канала вне записей бота не рассматриваются вовсе.

**Напоминания.** Зрителю — за `RENEW_NOTICE = 7 суток` до конца оплаченного окна (продление = новый эскроу тем же флоу). Владельцу — при наличии созревших несобранных кусков: «подписчики вправе вернуть несобранное через `refund()` спустя `RELEASE_MARGIN`».

Все константы §3–§5 — политика бота: правятся конфигом, каменеть им не нужно.

## 6. Транзакционные флоу (мини-апп)

Все транзакции подписывает кошелёк пользователя; газ платит подписант. Бот в этих флоу не участвует — мини-апп говорит с канистрами (`@dfinity/agent`, анонимно — методы бесправны) и цепью (RPC из конфига) напрямую.

**Подписаться.** Параметры канала (резолвер, `owner_wallet`, `price`, `period`) мини-апп получает из диплинка/бота, донор выбирает `n_chunks` ≥ 1; `t0 = now`; `nonce` — случайный u64. Адрес эскроу — `PDA([b"escrow", salt], factory_stream)`, соль — `core/`-порт `crown_salt::stream` (§8). Инструкция `create_escrow` — по IDL формы; кусок 0 созревает сразу.

**Собрать (владелец).** Список эскроу канала — `getProgramAccounts(factory_stream)` с memcmp-фильтром `resolver @ offset 72` (раскладка аккаунта `Escrow` формы: `donor@8..40`, `salt@40..72`, `resolver@72..104`); созревшие куски: `released < n_chunks ∧ now ≥ t0 + released·period ∧ !settled`. На каждый — `subscription.request_release` → пара инструкций `[ed25519_program(подпись), release(index)]`; пары пакуются в транзакции, пачка подписывается одним `signAllTransactions`. Строгий порядок индексов внутри одного эскроу — последовательными инструкциями/транзакциями.

**Отменить (донор).** Мини-апп восстанавливает адрес эскроу, донор подписывает сообщение отмены канистры (`"crown:subscription:v1" ‖ lp(chain) ‖ lp(canister_id) ‖ lp(escrow) ‖ 0x00` — раскладка game-spec «Подписки» §8) → `request_cancel` → транзакция `[ed25519_program, cancel]`. Остаток мгновенно у донора; бот увидит `settled` на ближайшей ревизии.

**Страховка.** `refund()` доступен из мини-аппа любому, когда форма позволит; подписи не требует.

Эскроу зрителя для проверок и карточек ищутся тем же `getProgramAccounts` с memcmp `donor @ 8` + `resolver @ 72`.

## 7. Хранение

SQLite, один файл. Хранится **только то, чего нет в цепи**:

```
channels:  channel_id PK, tg_chat_id, owner_wallet, owner_telegram_id,
           setup_nonce, resolver, price, period, threshold, created_at
bindings:  channel_id, wallet, telegram_id, bound_at, unbound_at NULL,
           cooldown_until            UNIQUE(channel_id, wallet) среди активных
members:   channel_id, telegram_id, admitted_at, admitted_by ('reputation'|'subscription'),
           threshold_at_admission, grace_started_at NULL, kicked_at NULL
```

В памяти процесса: выданные челленджи (nonce → telegram_id, channel_id, expires).

Не хранится никогда: эскроу, балансы, репутация, статусы подписок, история платежей — всё это читается из первоисточников на каждой проверке. Потеря базы: привязки и политики переустанавливаются заново; ничего денежного не существует в базе by construction. Мэппинг `telegram_id ↔ wallet` — деанонимизирующие данные; их минимальность и локальность на VPS — принятое ограничение (§10.3).

## 8. Байт-критичное и чтение первоисточников

**Соль `stream` на TS** — порт формулы `crown_salt::stream` (третья реализация после ончейн и Rust-крейта). Эталон — фикстуры-копии `vectors/stream-salt.json` из `crown-factory` (там они кросс-проверены python ↔ Rust ↔ ончейн-фазз); юнит-тест `core/` гоняет все вектора. PDA — `PublicKey.findProgramAddressSync`, не самописный.

**Инструкции формы** — дискриминаторы Anchor (`sha256("global:<имя>")[..8]`) пиннутся тестами; аккаунт-листы — по задеплоенной программе. Раскладка аккаунта `Escrow` (offsets §6) пиннута тестом на реальном devnet-аккаунте (фикстура).

**Сообщения канистры и челлендж** — раскладки §3 и game-spec «Подписки» §8, каждая в одном модуле `core/`, пиннуты векторными тестами.

**Чтение.** Цепь — JSON-RPC из конфига (недоверенная зона, любой провайдер; `commitment = confirmed` для UI, `finalized` — для ревизии). Книга и канистра — plain query/update без верификации сертификатов в v1 (§10.4). `getProgramAccounts` требует RPC-провайдера с включённым методом — фиксируется в конфиге.

## 9. Конфиг

```toml
[chain]
id             = "solana-devnet"
rpc            = "<url>"                        # недоверенная зона: свой RPC разрешён
factory_stream = "<program id>"                 # из crown-factory deploy/
usdc           = "<mint>"

[canisters]
subscription = "<principal>"                    # crown-games/subscription
crown_index  = "<principal>"                    # книга

[policy]                                        # дефолты, правятся свободно
challenge_ttl   = 300
rebind_cooldown = 259200
max_wallets     = 10
revision_period = 21600
grace           = 259200
renew_notice    = 604800
```

Токен бота — только env (`BOT_TOKEN`), путь к базе — env. Секретов в конфиге и репозитории нет. Адреса программ и канистр этот репозиторий не порождает — только копирует из деплоев.

## 10. Границы доверия и принятые ограничения

**Что бот может.** Не впустить, кикнуть, промолчать, наврать в карточке. Ровно те же полномочия есть у владельца канала руками — бот не добавляет новых.

**Чего бот не может.** Украсть или задержать деньги (их у него нет), подделать репутацию (книга сертифицирована и пишется только сплиттером), помешать донору отменить подписку или дождаться `refund()` (мини-апп + канистра работают без бота), выпустить кусок себе (подпись release исполнима только в пользу вшитых при рождении получателей).

Принятые ограничения — не латать, а принять:

1. **Telegram — платформенный риск.** Канал, бот и токен живут по правилам Telegram; бан или изменение Bot API останавливают членский менеджмент, но не деньги. Это осознанный перенос хостинга контента и модерации на платформу.
2. **Оператор бота видит привязки.** `telegram_id ↔ wallet` — деанонимизирующий мэппинг, необходимый для ревизии. Минимизирован (§7), но существует; самохост фрагментирует его по владельцам (post-bot).
3. **Пересылка контента наружу не решается.** Как и везде.
4. **Plain-чтения в v1.** Ответы книги и RPC не верифицируются криптографически; злонамеренный провайдер может обмануть бота о членстве — вред ограничен полномочиями §10.1. Сертифицированные чтения — post-bot.
5. **Складчина кошельков** на один аккаунт возможна: одно место на пул, кошельки заперты кулдауном. Принято (§3).
6. **Владелец-кит.** Владелец может добавить кого угодно руками мимо всей механики — это его канал. Бот не оспаривает.

## 11. Обновление

Бот и мини-апп деплоятся свободно и навсегда: недоверенной зоне заморозка не положена. Обновление канистры подписки (новый principal) = новая запись конфига; каналы пере-сетапятся на новый резолвер, старые эскроу доживают у старого. Схема SQLite версионируется миграциями при старте сервиса.
