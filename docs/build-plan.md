### B0 — Скелет: три части, границы, CI

**Выход.** Воркспейс npm: `core/`, `tg/`, `miniapp/`. `config/testnet.toml`, `config/mainnet.toml` по схеме bot-spec §9. CI: typecheck, lint, test, и структурные линты ниже.

**Структурные линты (это и есть суть этапа).**
1. `core/` не импортирует Telegram, SQLite, Node-специфичные API: `grep -rE "grammy|better-sqlite3|node:" core/src/` → пусто.
2. `tg/` не подписывает и не шлёт транзакции: `grep -rE "Keypair|signTransaction|sendTransaction|sendRawTransaction" tg/src/` → пусто.
3. Секретов в репозитории нет: `grep -riE "BOT_TOKEN\s*=|[0-9]+:[A-Za-z0-9_-]{35}" --include='*.ts' --include='*.toml'` → пусто; токен — только env.
4. `miniapp/` не читает env и не содержит секретов: `grep -rE "process\.env" miniapp/src/` → пусто.
5. Зависимости в одну сторону: `tg → core`, `miniapp → core`; импорт `tg` из `miniapp` и обратно — падение линта.

**DoD.** Дерево совпадает с картой в `CLAUDE.md`. Линты зелёные на пустом репозитории. Ни одного адреса, program id или principal в коде — только `config/`.

### B1 — `core/` — байт-критичное и чтение первоисточников

**Вход.** bot-spec §2, §3, §6, §8.

**Выход.** Библиотека: соль `stream` (TS-порт), адрес эскроу (PDA через web3.js), `channel_id`, челлендж §3, сообщение отмены канистры, декодер аккаунта `Escrow`, предикаты `subscription_alive`/`entitled` (чистые функции от переданных данных), клиенты чтения (RPC: `getProgramAccounts` по memcmp `donor@8`/`resolver@72`, `getAccountInfo`; agent-js: `get_reputation`, `get_resolver`, `request_release`, `request_cancel`), сборщики инструкций (`create_escrow`, `ed25519_program` + `release`/`cancel`, `refund`).

**DoD.**
- Соль: все вектора фикстуры-копии `vectors/stream-salt.json` (из `crown-factory`) проходят; копия побайтово сверена с оригиналом скриптом CI.
- `channel_id`, челлендж, сообщение отмены — пиннуты кросс-инструментальными векторами (python3 hashlib / фиксированные байты).
- Декодер `Escrow`: смещения `donor@8`, `salt@40`, `resolver@72` и хвост полей пиннуты фикстурой реального devnet-аккаунта.
- Дискриминаторы инструкций пиннуты (`sha256("global:<имя>")[..8]`).
- Предикаты `entitled`/`subscription_alive` покрыты табличными тестами: все ветви закона bot-spec §4, включая «доступ не зависит от `released`», грани окна, `settled`.
- Ни одного сетевого вызова в юнит-тестах: клиенты чтения тестируются на фикстурах ответов.

### B2 — Мини-апп: кошелёк и три флоу

**Вход.** bot-spec §3 (транспорт, челлендж), §6.

**Выход.** Статическая страница: подключение кошелька (wallet-adapter), SIWS-подпись челленджа и возврат через `sendData`, флоу «подписаться» (выбор `n_chunks`, `create_escrow`), «собрать» (обнаружение созревших кусков канала, `request_release`, пачка `signAllTransactions`), «отменить» (подпись сообщения отмены, `request_cancel`, транзакция). Деплой в asset-канистру (локальная реплика для разработки).

**DoD.**
- Против devnet и живой канистры подписки, руками по чек-листу: создать эскроу (короткий период) → собрать два созревших куска кошельком владельца одной пачкой → отменить остаток кошельком донора; балансы и `Settled` сверены эксплорером; книга локального `crown-index` показывает донору `+Σ piece`.
- Строгий порядок индексов при сборе нескольких кусков одного эскроу соблюдён (последовательные транзакции).
- `sendData`-путь отдаёт боту ровно `(wallet, signature, nonce)`; размер < 4096 байт.
- Страница не делает ни одного запроса к сервису бота: только цепь и канистры.

### B3 — Сервис: привязки, допуск, ревизия

**Вход.** bot-spec §3, §4, §5, §7.

**Выход.** Сервис `tg/`: long polling; `/setup` владельца (права, политика, `channel_id`, резолвер, диплинк); приём `web_app_data` (верификация `initData` HMAC, проверка челленджа, запись привязки с инвариантами и кулдауном); обработка заявок на вступление (`entitled` → одобрить; нет привязки → оставить владельцу с пометкой); ревизия по таймеру (grace, кик только своих); напоминания (продление, несобранные куски). SQLite-схема §7 с миграциями.

**DoD.**
- Интеграционные тесты с моком Bot API и фейковыми часами: полный сетап; привязка (валидная, просроченный челлендж, повторный nonce, чужой `telegram_id` — отвергнуты); перепривязка кикает старого и включает кулдаун; два аккаунта на один кошелёк невозможны; `MAX_WALLETS` соблюдён.
- Допуск: репутационный вход (порог на момент входа фиксируется), подписочный вход, «оба нуля» невозможны с сетапа.
- Ревизия: потеря права → предупреждение → grace → кик; участник вне записей не тронут никогда — негативный тест; неопознанная заявка не отклонена.
- Первоисточники в тестах — фикстуры B1; ни одного живого сетевого вызова в CI.
- Рестарт сервиса теряет только незавершённые челленджи; база цела (тест на миграции).

### B4 — Сквозной прогон: живой канал на devnet

**Вход.** вся спека.

**Выход.** Ручной e2e-сценарий (чек-лист в `scripts/e2e-checklist.md`) на реальном приватном канале Telegram, devnet, живая канистра подписки, локальный `crown-index`.

**DoD.**
- Сетап реального канала; вход по подписке (создание эскроу из мини-аппа → заявка → автоодобрение); сбор кусков владельцем; отмена донором → ревизия (сжатые интервалы) → предупреждение → кик.
- Вход по репутации: кошелёк с накопленной devnet-репутацией ≥ порога впущен без подписки.
- Перепривязка кошелька на второй Telegram-аккаунт кикает первый.
- Ручной участник, добавленный владельцем, переживает все ревизии.
- Смерть сервиса (останов процесса) не мешает отмене из мини-аппа — проверено при остановленном боте.
- `git diff` в `crown-core`, `crown-factory`, `crown-games/*` пуст.

После зелёного B4 бот готов к тихому прод-запуску на devnet-экономике; mainnet — вместе с платформой (S4/F6/G5 соответствующих репозиториев), для бота это смена профиля конфига.
